Feedback Loop

Feedback Repair Loop

这份文档解释当前 feedback 模块的职责：当 Agent 借助 raw 工具兜底后，系统如何把这些信号沉淀成修复任务、候选补丁和人工编审流。

1. 这个模块解决什么问题

Agent 读 raw 往往意味着 wiki 不够。feedback 模块的目标就是把这种“问答时临场兜底”变成“后续可以修 wiki 的任务”。

2. FeedbackTask 保存了什么

  • 来源对话内容
  • 工具轨迹、wiki reads、raw reads
  • evaluator 结果和置信度
  • repair candidate
  • review guidance 和 revision_count

所以它不是一个单纯的“待办事项”，而是完整的审计对象。

3. 状态流转

当前状态集合包括：

  • pending_evaluation
  • evaluation_done
  • pending_review
  • pending_recompile
  • approved
  • rejected
  • applied
  • compile_failed

状态迁移是受限的，不是任意跳转。

4. 人工编审不是纯手工改文档

当前 review 动作有三种：

  • approve
  • reject
  • revise

revise 的重点是用户输入一段一次性 guidance，系统随后触发重编译，而不是要求用户直接手改 markdown。

5. 去重与限制

  • 同一页面在一定时间窗内支持重复任务去重。
  • 每个任务的 revision 次数受 feedback.max_revisions 限制。
  • 模型配置与主 LLM 配置独立。
