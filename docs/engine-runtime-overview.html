Engine Runtime Overview

这份文档从系统整体视角解释 llm-wiki-engine 怎么运行：应用启动时做什么、项目和磁盘目录怎么关联、文档上传后怎么进入 ingest、wiki/search/chat/agent/feedback 这些模块之间的边界是什么。

1. 启动时会发生什么

  1. FastAPI 应用启动，执行 app/main.py 里的 lifespan。
  2. 调用 init_db() 初始化数据库、补 migration、加载系统配置覆盖。
  3. 启动 ingest 队列 worker。
  4. 恢复中断的 ingest job。

也就是说，engine 的运行不是“纯 stateless API”。它在进程内维护了一个 ingest 队列执行器，并且依赖磁盘项目目录持续存在。

2. 系统的几个核心对象

Project系统的基础单位。一个项目同时对应数据库记录和磁盘目录。
Raw Documents上传到 raw/sources/ 的原始资料，是 ingest 的输入。
Wiki编译后的 markdown 页面，位于 wiki/ 目录，是检索、图谱、普通 chat 的主要知识源。
Agent面向工具调用的问答入口，可跨项目、可读 raw、可选调用 ticket wiki。
Feedback Task回答后异步产生的修复任务，用于把 raw 兜底信号变成 wiki 修复候选。

3. 主运行链路

用户/前端
  -> 项目创建
  -> 上传 raw 文档
  -> 触发 ingest
  -> 生成 wiki/*.md + embedding + graph
  -> 普通 search / chat 使用 wiki
  -> Agent 按需 search wiki / read wiki / read raw / search ticket
  -> 如果 Agent 依赖 raw 完成回答
  -> 进入 feedback queue，准备后续修复 wiki

4. 每个模块的职责边界

  • projects/：创建项目、维护成员权限、初始化磁盘目录。
  • documents/：把上传文件保存到 raw/sources/。
  • ingest/：把 raw 文档编译成 wiki 页面，并触发 embedding。
  • wiki/：暴露文件树、页面读写、overview、graph。
  • search/：关键词检索、向量检索、融合检索。
  • chat/：普通项目聊天，走预拼上下文 RAG。
  • agents/：toolcall orchestrator，支持 wiki/ticket/raw 工具。
  • feedback/：异步修复闭环和人工编审流。
  • admin/：在线修改 LLM / embedding / feedback 模型设置。

5. 为什么这个系统是“磁盘优先”的

当前 engine 的知识主存不是数据库，而是项目目录本身。数据库更多是做索引和权限：

  • 项目页面实际存储在磁盘的 wiki/ 里。
  • 聊天记录存储在 .llm-wiki/chats/ 里。
  • embedding 存储在 .llm-wiki/lancedb/ 里。
  • 数据库里只存项目、用户、Agent、Job、FeedbackTask 等元信息。

这也是为什么理解项目目录结构对维护 engine 非常重要。

6. 普通 Chat 和 Agent 的分叉点

两条链路的核心区别是：

  • 普通 chat：后端先做检索和拼接，然后把相关 wiki 页面直接塞进 prompt。
  • Agent：后端只给模型工具，模型自己决定要不要搜 wiki、读页、查 ticket、读 raw。

所以普通 chat 更像固定 RAG，Agent 更像受控的工具编排运行时。
