Search And Retrieval

这份文档解释 engine 当前的检索体系：关键词检索、向量检索、RRF 融合、图谱扩展，以及 embedding 是如何支撑向量检索的。

1. 检索输入是谁

  • 普通搜索 API：用户显式输入查询。
  • 普通 chat：直接用用户消息作为查询。
  • Agent tool：模型生成 search_wiki / search_ticket_cases 的查询参数。

当前没有统一的 query rewriting 层，绝大多数时候就是“拿用户或模型给出的 query 直接搜”。

2. 关键词检索

search_bm25() 会扫描项目下的 wiki 页面，做 token 化和 BM25 打分，并给出标题、path、snippet。

它的优势是：

  • 精确术语命中稳定
  • 文件名、标题短语命中容易拿高分
  • 不依赖 embedding 服务

3. 向量检索

向量检索依赖每个项目自己的 .llm-wiki/lancedb/wiki_chunks_v2 表。

  1. 页面去 frontmatter 后按 markdown chunker 切块。
  2. 每个 chunk 用 embedding 模型得到向量。
  3. 写入 LanceDB，保留 page_id、chunk_index、heading_path、chunk_text。
  4. 搜索时对 query 也做 embedding，然后查近邻 chunk。

4. 融合检索

默认模式是 hybrid：BM25 和向量结果各取一批，再交给 rrf_fusion() 做 Reciprocal Rank Fusion。

这样做的目标是：

  • 兼顾精确名词命中与语义相似
  • 避免只靠 BM25 时对口语问法太脆
  • 避免只靠向量时丢掉标题/文件名的强信号

5. 图谱扩展只在普通 chat 中自动启用

普通 chat 在拿到融合结果后，会用知识图谱做 1-hop 扩展，把相关页面再带进上下文。Agent 当前不会自动做这一步，而是由模型自己决定要不要继续读别的页。

6. Embedding 更新时机

  • ingest 写完 wiki 文件后会调用 embed_pages()
  • wiki 页面被 API 修改后也会尝试重新 embedding 该页

也就是说，embedding 不是离线批任务，而是跟随 wiki 变更局部更新。
