Agent Runtime

Agent Tool Runtime

这份文档解释 Agent 在 engine 里是如何运行的：工具集合怎么注册、tool context 里放了什么、orchestrator 如何循环执行，以及调试轨迹如何生成。

1. ToolContext

每次 agent turn 都会构建一个 ToolContext，里面最关键的只有两个东西：

  • main_projects：Agent 当前可访问的主项目集合
  • ticket_project：若某个主项目绑定了案例库，则加载这个 ticket 项目

工具定义是否包含 ticket 能力，完全由这个上下文决定。

2. 当前工具集合

  • search_wiki
  • read_wiki_page
  • get_wiki_index
  • get_project_purpose
  • read_raw
  • grep_raw
  • search_ticket_cases（按上下文可选）
  • read_ticket_page（按上下文可选）

3. Orchestrator 循环

  1. 构建系统提示，里面写明工具使用策略。
  2. 把历史消息和当前用户消息组成 messages。
  3. 调用 complete_with_tools()。
  4. 如果模型返回 tool calls，就逐个执行工具。
  5. 把工具结果作为 role=tool 消息回填。
  6. 继续下一轮，直到模型只输出文本不再调用工具。

4. 预算与保护

  • 有 max_tool_calls 上限，超过后会拒绝新的工具调用并要求模型直接收敛回答。
  • 长工具结果会被 debug_result_limit 截断，只影响 SSE 调试输出，不影响真正回填给模型的结果。
  • read_raw 只允许读 raw/sources/，且有最大行数限制。

5. 为什么这套 runtime 值得单独理解

因为它不是普通 chat 的延伸，而是一套受控工具执行环境。你改工具 schema、路径保护、结果截断、ticket 策略、raw 读取范围，都会直接改变 Agent 的行为边界。
