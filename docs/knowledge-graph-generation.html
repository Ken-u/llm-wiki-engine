LLM Wiki Engine Docs

Knowledge Graph Generation

本文档解释当前 llm-wiki-engine 里的知识图谱数据是如何从 wiki/ 目录下的 markdown 文件动态推导出来的。重点不是 API，而是： 一个 wiki 页面怎样变成 node，哪些内容会变成 edge，页面标题、类型、路径、邻居关系分别来自哪里，以及当前实现有哪些约束。

一句话结论： 当前知识图谱不是单独编译产物，也不存数据库；它是在请求图谱页面时，实时扫描项目下所有 wiki/**/*.md 文件，再根据 frontmatter 的 related 和正文里的 [[wikilink]] 关系构建出来的。

1. 整体思路

当前图谱逻辑的核心实现位于 app/wiki/graph.py 的 build_wiki_graph(project_dir)。它做的事情很直接：

  1. 递归扫描项目下的 wiki/ 目录，找到所有 .md 页面。
  2. 每个页面先生成一个 graph node。
  3. 再从两个地方抽取关系：frontmatter 的 related，以及正文里的 [[wikilink]]。
  4. 把这些关系存成 edge，并顺手给 node 填好 neighbors。
这意味着知识图谱和 wiki 文件是强绑定的：如果你改了某个 markdown 页面的标题、类型、related、wikilink， 图谱下次请求时就会跟着变，不需要额外编译步骤。

2. 扫描范围

图谱只扫描项目目录下的 wiki/ 子树，也就是： 

<project>/
  wiki/
    overview.md
    index.md
    entities/
    concepts/
    sources/
    queries/
    comparisons/
    synthesis/
    ...

只要文件后缀是 .md，无论在 wiki/ 根下还是任意子目录下，都会被当成一个候选页面。 raw/、purpose.md、schema.md 这类非 wiki/ 文件不会进入图谱。

3. Node 是怎么来的

每个 wiki/**/*.md 文件都会先生成一个 node。当前实现里，node 主要有 4 个核心字段：

id 文件名去掉扩展名后的 stem。例如 wiki/entities/gms.md 的 node id 是 gms。
title frontmatter 里的 title；如果没写，就回退成 id。
type frontmatter 里的 type，例如 entity、concept、source。
path 相对项目根目录的路径，例如 wiki/entities/gms.md。

这些字段通过 parse_frontmatter() 读取 YAML frontmatter 后组装到 node 中。

4. Edge 是怎么来的

当前 edge 只有两种来源，没有第三种：

  1. 页面 frontmatter 里的 related
  2. 页面正文里的 [[wikilink]]

当前实现不会从 tags、sources、目录层级、标题相似度、embedding 相似度中自动生成边。 所以图谱连接强度完全取决于页面作者或编译器是否明确写下了这些关系。

5. related 的映射规则

如果页面 frontmatter 里有这样的内容：

---
type: entity
title: GMS
related:
  - google-mobile-services
  - mada-edla
---

那么图谱会直接加两条边： 

(当前页 page_id, "google-mobile-services")
(当前页 page_id, "mada-edla")

这里不会做进一步解析。也就是说，related 里写进去的值会被直接当成目标 page id 使用。

这点很重要：related 不是 path，也不是 title 匹配。 当前实现默认你写进去的就是目标页面的“图谱 id”。

6. [[wikilink]] 的映射规则

正文中的 wikilink 先用一个非常简单的正则提取： 

\[\[([^\]]+)\]\]

提取到的 target 会再做 3 步归一化：

  1. strip() 去首尾空白
  2. lower() 转小写
  3. 把空格替换成 -

例如：

[[GMS]]                    -> gms
[[MADA EDLA 协议]]         -> mada-edla-协议
[[Google Mobile Services]] -> google-mobile-services

然后图谱直接创建一条边： 

(当前页 page_id, 归一化后的 target)

这里没有更复杂的解析逻辑，不会去查 path，也不会用前端那套 title/path/slug 多候选匹配规则。

7. 邻居关系怎么算

图谱内部先累积原始 edges，然后做第二遍处理来填每个 node 的 neighbors：

  1. 如果有边 (src, dst)，则把 dst 加到 src.neighbors
  2. 如果目标节点 dst 真的存在，也会把 src 加到 dst.neighbors
  3. 最后对每个 node 的 neighbors 去重并排序

所以从 JSON 形态上看，edges 是原始方向边，neighbors 更像“图上可达的相邻节点列表”。

8. 返回给前端的数据结构

图谱最终会被序列化成这样一类结构：

{
  "nodes": [
    {
      "id": "gms",
      "title": "GMS",
      "type": "entity",
      "path": "wiki/entities/gms.md",
      "neighbors": ["google-mobile-services", "mada-edla"]
    }
  ],
  "edges": [
    {
      "source": "gms",
      "target": "google-mobile-services"
    }
  ]
}

前端图谱页不会再去推理这些关系，它只拿到这份 JSON 后做可视化布局和点击预览。

9. 一个完整例子

假设项目里有两个页面：

wiki/entities/gms.md
wiki/concepts/mada-edla.md

wiki/entities/gms.md 内容：

---
type: entity
title: GMS
related:
  - mada-edla
---

如果未签署 [[MADA EDLA]]，则无法直接获取 GMS 包。

wiki/concepts/mada-edla.md 内容：

---
type: concept
title: MADA EDLA
---

生成结果会是：

  • node 1：id=gms，title=GMS，type=entity，path=wiki/entities/gms.md
  • node 2：id=mada-edla，title=MADA EDLA，type=concept，path=wiki/concepts/mada-edla.md
  • edge A：来自 related，即 (gms, mada-edla)
  • edge B：来自正文 [[MADA EDLA]]，归一化后也是 (gms, mada-edla)

注意这里最后会出现两条原始 edge，但 neighbors 在 node 上会去重。

10. 与 wiki 文件格式的对应关系

如果从“写 wiki 文件的人”角度看，当前知识图谱只关心这几类内容：

title 决定图上的显示标题。
type 决定节点类型，前端会据此着色。
related 显式声明页面之间的图谱边。
[[wikilink]] 从正文里再推导一层页面关系。

其他 frontmatter 字段，例如 tags、sources、created、updated， 当前都不会参与图谱连边。

11. 当前实现的限制与风险

  1. node id 只取文件名 stem。 如果 wiki/entities/gms.md 和 wiki/queries/gms.md 同时存在， 两个页面都会争用 gms 这个 id，后扫描到的页面会覆盖先前页面。
  2. wikilink 解析很原始。 只支持简单的 [[...]]，不支持更完整的 title/path/alias 容错逻辑。
  3. related 依赖人工写对目标 id。 如果 frontmatter 里写的是 title 或 path，而不是目标页面的 stem/slug，图谱边可能连不上真实节点。
  4. 可能存在悬空边。 edge 可以指向一个并不存在的目标 id；这种边会保留在 edges 里，但不会生成完整的目标 node。
  5. 没有缓存。 每次请求都会重新扫 wiki/，项目很大时会增加实时开销。
所以当前图谱更像“基于 markdown 语法和 frontmatter 的轻量页面关系图”，而不是一个经过规范化建模、去重、别名解析后的严格知识图谱。

12. 代码位置

  • app/wiki/graph.py：图谱扫描、node/edge 构建、JSON 序列化
  • app/wiki/frontmatter.py：frontmatter 解析，提供 title、type、related 等字段
  • app/wiki/router.py：图谱页面请求时触发构建
  • src/components/wiki/KnowledgeGraph.tsx：前端消费图谱 JSON 并做画布渲染

文档基于当前代码实现整理。如果后续统一了 wikilink 解析逻辑、改成 path-based node id 或加入缓存，本页结论也需要同步更新。
