Runtime Split

Chat And Agent Runtime

这份文档专门解释两个问答运行时：普通项目 chat 和 Agent chat。它们都回答问题，但知识接入方式完全不同，维护时必须分开理解。

1. 普通项目 Chat：固定 RAG

  1. 从 .llm-wiki/chats/ 和 conversations.json 读取会话历史。
  2. 对用户问题执行 hybrid search：BM25 + vector + RRF。
  3. 根据命中的 page id 做 1-hop 图谱扩展。
  4. 按上下文预算读取 purpose、index、相关 wiki 页面。
  5. 拼成一个 system prompt，然后直接流式调用 LLM。
  6. 回答完成后，把 user/assistant 消息写回磁盘聊天记录。

这里模型没有显式调用工具的能力，所有知识都是后端先准备好的。

2. Agent Chat：工具编排

  1. 加载 Agent 配置和关联项目。
  2. 构建 ToolContext：主项目列表 + 可选 ticket 项目。
  3. 根据上下文动态注册工具定义。
  4. 进入 run_agent_turn() 循环，调用 complete_with_tools()。
  5. 如果模型发起 tool call，就执行工具并把结果回填到消息流中。
  6. 如果模型不再调用工具，则输出最终回答。

这里知识不再由后端一次性预装，而是由模型在运行时决定什么时候搜、什么时候读页、什么时候读 raw。

3. 二者的根本差异

  • 普通 chat 的知识源固定：主项目 wiki。
  • Agent 的知识源可按需切换：主 wiki、ticket wiki、raw/sources。
  • 普通 chat 的预算主要是“后端预先拼多少内容”。
  • Agent 的预算主要是“最多允许多少次工具调用、每次工具返回多大内容”。

4. 为什么两条链路都还保留

当前系统没有把普通 chat 也迁到全 toolcall。原因很简单：

  • 普通 chat 更稳定，成本和时延更可控。
  • Agent 更灵活，适合复杂问题、跨项目问题、需要 raw 兜底的问题。

所以维护时不要假设“chat 和 agent 只是 UI 不同”，它们是两套知识接入架构。
