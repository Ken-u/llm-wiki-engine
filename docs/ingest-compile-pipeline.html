Wiki Compile / Ingest Pipeline

本文档描述 llm-wiki-engine 当前实际运行的 wiki 编译流程。这里的“编译”就是把 raw/sources/ 里的原始文档解析、送入 LLM、生成 wiki/ 下的结构化页面，并重新做 embedding；同时标出哪些步骤会读写 SQL 业务库或 LanceDB 向量库。

这份说明基于当前代码实现整理，不是未来规划。重点回答几个问题：

  • 编译入口在哪里

  • 原始文档如何进入 prompt
  • 大文档如何避免上下文爆炸
  • LLM 输出如何被解析、落盘、合并
  • 哪些步骤会和数据库交互
  • 失败后如何重试和恢复

1. 结论先说

当前编译流程不是 Agent，也没有使用 toolcall 编排。

它是一条固定的后端流水线：

  1. 读取源文档并解析成文本
  2. 做缓存判断
  3. 调用 LLM 做第 1 步分析
  4. 调用 LLM 做第 2 步生成
  5. 解析 ---FILE: ... --- 输出块
  6. 把页面写入 wiki/
  7. 对已有页面做 frontmatter 合并，必要时再调用一次 LLM 合并正文
  8. 重新生成 embedding

换句话说，当前 ingest 更接近“大 prompt 编译器”，而不是“模型自主决定下一步做什么”的 Agent。

4. API 层如何触发编译

入口在 POST /api/projects/{project_id}/ingest，实现位置是 app/ingest/router.py。

  • 先通过数据库校验用户是否属于项目：check_membership(db, project_id, user)
  • 再从数据库读取项目记录：get_project_or_404(db, project_id)
  • 使用项目记录里的磁盘目录 project.disk_path
  • 默认扫描 <project>/raw/sources/ 下的所有源文件
  • 如果请求里带了 source_file，则只编译该文件
  • 每个源文件调用 ingest_queue.enqueue()，在数据库里创建一个独立 ingest_jobs 记录
  • API 返回 202 Accepted，实际处理异步执行

补充接口：

  • POST /api/projects/{project_id}/ingest/{job_id}/retry：查询旧 job，校验项目归属，把旧 job 标记为 superseded 后创建新 job
  • GET /api/projects/{project_id}/ingest/status：查询数据库中 pending / processing 的 job
  • GET /api/projects/{project_id}/ingest/history：查询非 superseded 的历史 job

5. Job 和队列模型

任务模型在 app/ingest/models.py 的 IngestJob。

source_path源文件的绝对或项目内路径
statuspending / processing / done / failed / superseded
progress当前进度文案，面向 UI 展示
step0=queued 1=analysis_done 2=generation_done 3=written
files_written本次最终写入的 wiki 文件路径列表
error失败原因或最近一次重试错误
retry_count当前重试次数
created_by触发该 job 的用户 ID

队列在 app/ingest/queue.py，关键行为如下：

  • 同一项目串行执行，避免多个任务同时改 wiki/index.md 或 wiki/overview.md
  • 不同项目可并行
  • 服务启动时会恢复数据库里处于 pending 或 processing 的任务
  • MAX_RETRIES = 2，失败自动重试，且从上次完成 step 继续

6. 数据库交互边界

当前 ingest 流程涉及两类数据库：全局 SQLAlchemy 业务库，以及每个项目目录下独立的 LanceDB 向量库。wiki 页面正文、编译缓存和 checkpoint 都不是 SQL 记录，而是项目目录里的文件。

SQL 业务库Project、ProjectMember、IngestJob 等表，使用 AsyncSession
LanceDB 向量库.llm-wiki/lancedb/wiki_chunks_v2，保存 wiki chunk 和向量
文件系统状态wiki/、ingest-cache.json、checkpoints/，不进 SQL

SQL 交互点如下：

  • 触发编译时，API 查询 ProjectMember 做权限校验，查询 Project 获取 disk_path。
  • enqueue() 插入 ingest_jobs：初始状态为 pending，step=0，记录 source_path 和 created_by。
  • 任务执行中，队列反复更新 ingest_jobs.progress 和 ingest_jobs.step，用于前端状态展示和断点恢复。
  • 任务成功后，队列更新 status=done、progress=Complete、step=3、files_written、completed_at。
  • 任务失败时，队列查询当前 job 的 retry_count 和 step；未超过重试上限则更新回 pending，否则更新为 failed。
  • 服务启动时，recover_interrupted_jobs() 查询 pending / processing 的 IngestJob，再查询对应 Project.disk_path 重新入队。
  • 状态、历史、删除、重试接口都会查询或修改 ingest_jobs；删除接口会对非运行中的 job 执行 db.delete(job)。

LanceDB 交互点只有 embedding 阶段：embed_pages() 连接 .llm-wiki/lancedb，打开或创建 wiki_chunks_v2 表，按 page_id 删除旧 chunk，再写入本次页面的新 chunk 向量。

7. 主流程总览

主函数是 app/ingest/pipeline.py 中的 auto_ingest()。

Parse document
  -> Cache check
  -> Load checkpoint
  -> Read project context
  -> Truncate source text
  -> Step 1: Analysis
  -> Step 2: Generation
  -> Parse FILE blocks
  -> Write files
  -> Embed pages
  -> Update cache
  -> Remove checkpoint

8. 解析原始文档

原始文档解析在 app/documents/parser.py。

  • PDF: PyMuPDF
  • DOCX: python-docx
  • XLSX / XLS: openpyxl
  • TXT / MD / MARKDOWN / RST / CSV / JSON / YAML / YML: 直接按文本读取
  • 其他后缀：回退到按文本读取

如果解析后内容为空，任务会直接跳过，不会进入后续 LLM 阶段。

9. 缓存机制

缓存逻辑在 app/ingest/cache.py。

  • 对源文档内容做哈希
  • 以 source_identity = 源文件名 作为索引键
  • 落到 .llm-wiki/ingest-cache.json

命中缓存时当前文件不会重新分析和生成，旧 checkpoint 会被删除，auto_ingest() 直接返回空列表。

注意：如果只是 prompt 或生成逻辑变化，而源文档没变，缓存会导致跳过，需要手动处理缓存或变更源内容。

10. Checkpoint 机制

checkpoint 文件位于：

.llm-wiki/checkpoints/<sha256(source_path)前16位>.json

存储内容包括 step、source、analysis、generation。

主流程会比较数据库中的 job.step 和 checkpoint 中的 step，取两者最大值作为 effective_step，用于从中间状态恢复。

11. 编译前读入哪些上下文

在真正调 LLM 之前，系统会读取这些项目文件：

  • purpose.md
  • wiki/index.md
  • wiki/overview.md
  • schema.md

这些文件分别用来约束目标、告知现有页面结构、更新 overview，以及约束生成格式。

12. 原始文档如何进入 prompt

当前实现里，解析出来的原始文本会直接拼进 prompt，但会先做截断：

max_content_chars = 100_000
truncated = content[:max_content_chars] if len(content) > max_content_chars else content
  • 当前不是按 chunk 多轮处理
  • 也不是检索式编译
  • 而是把文档前 100000 个字符作为 source context 直接送给模型
代价：大文档后半段可能根本不会进入模型上下文，这也是当前编译质量瓶颈之一。

13. Step 1: Analysis

analysis prompt 在 app/ingest/prompts.py 的 build_analysis_prompt() 中构造。

输入包括：

  • purpose.md
  • wiki/index.md
  • 截断后的 source_content
  • 当前源文件名

模型被要求分析 key entities、key concepts、main arguments、与现有 wiki 的关系、潜在矛盾和建议更新项。

这一阶段输出只是普通文本，不是结构化 JSON，也不是 tool result。

语言规则由文档前 2000 字的 CJK 比例启发式判断，中文比例高则强制中文输出，否则要求英文输出。

14. Step 2: Generation

generation prompt 在 app/ingest/prompts.py 的 build_generation_prompt()。

输入包括：

  • schema.md
  • purpose.md
  • wiki/index.md
  • wiki/overview.md
  • 源文件名
  • step 1 的 analysis
  • 截断后的原始文档文本

模型被要求一次性生成：

  1. 一个 source summary 页面
  2. 若干 entity 页面
  3. 若干 concept 页面
  4. 更新后的 wiki/index.md
  5. 追加到 wiki/log.md 的 log entry
  6. 更新后的 wiki/overview.md

默认 source summary 路径为：

wiki/sources/<源文件名去后缀>.md

输出必须严格遵守 FILE block / REVIEW block 格式，并带完整 YAML frontmatter。

15. 解析 LLM 输出

解析逻辑在 app/ingest/parser.py。

  • 识别 ---FILE: ... --- 和 ---END FILE---
  • 识别 ---REVIEW: ... --- 和 ---END REVIEW---
  • 自动关闭未闭合的 FILE block
  • 收集 warning
  • 拒绝不安全路径

只有 wiki/ 开头的相对路径会被接受。绝对路径、盘符路径、..、控制字符或非 wiki/ 路径都会被拒绝。

如果 generation 输出里一个合法 FILE block 都没有，本次任务不会写任何文件。

16. 写文件与页面合并

落盘逻辑在 app/ingest/writer.py。

  • 目标文件不存在时直接写入
  • wiki/log.md 特殊处理，总是 append
  • 已有页面会先合并 frontmatter，再决定正文如何处理

frontmatter 合并规则：

  • tags 取并集
  • related 取并集
  • sources 取并集
  • 其余字段默认新值覆盖旧值

wiki/index.md 和 wiki/overview.md 使用 incoming body 直接覆盖正文。

其他已存在页面会再发起一次 LLM merge，把旧页面和新页面合成一份。如果 merge 失败，则回退成“本次 incoming 正文 + 合并后的 frontmatter”。

17. Embedding 阶段

写完页面后，auto_ingest() 会调用 embed_pages(project_dir, written)。

  • 只对本次写入的页面做 embedding
  • 连接项目私有的 .llm-wiki/lancedb 向量库
  • 表名固定为 wiki_chunks_v2
  • 如果表已存在，会先按 page_id 删除该页面旧 chunk，再 add(records)
  • 如果表不存在，会按 schema 创建表并写入 records
  • embedding 失败是 non-fatal
  • 即使 embedding 报错，ingest 仍会完成并更新缓存

这意味着页面写盘和 embedding 不是一个事务，可能出现“页面已更新，但 embedding 暂时没更新”的短暂不一致。

18. 最终收尾

所有主要步骤成功后，系统会：

  1. 更新 .llm-wiki/ingest-cache.json
  2. 删除对应 checkpoint 文件

数据库中的 job 状态会被更新为：

  • status = done
  • progress = Complete
  • step = 3
  • files_written = [...]
  • completed_at = 当前时间

19. 重试和恢复的实际行为

失败自动重试时，队列会读取数据库里的 retry_count 和当前 job.step：

  • 若未超过 MAX_RETRIES，则状态改回 pending
  • progress 更新为 Retry x/y
  • 使用当前 job.step 作为 resume_step 继续执行

服务重启后，recover_interrupted_jobs() 会扫描数据库中处于 pending 或 processing 的任务，并结合项目磁盘路径重新入队。

20. 当前流程里实际有几次 LLM 调用

单个源文档最常见的情况是 2 次：

  1. analysis
  2. generation

如果命中了已有页面合并，则每个需要 merge 的已有页面还会额外再调一次 LLM。

21. 当前流程不是 Agent 的具体原因

虽然 app/llm/client.py 里已经有 complete_with_tools()，底层也具备 toolcall 能力，但 ingest 现在没有使用它。

  • 后端固定写死执行顺序
  • 模型不能自主决定下一步查什么、读什么、写什么
  • 没有 tool schema 暴露给模型
  • 没有多轮 tool orchestration
  • 没有运行时知识源调用决策

22. 当前实现的主要限制

大文档依赖截断只取前 100000 字符进入 prompt，超出部分直接忽略。
analysis 非结构化step 1 结果不是严格 schema 化的 facts。
generation 强依赖格式只要模型不按 FILE block 输出，后续就无法正常写盘。
merge 仍依赖 LLM已有页面合并不是 deterministic merge。
review block 没闭环当前不会把 REVIEW 转成真正的人工审阅任务。

23. 一次完整编译的时序图

User/API
  -> ingest router
     -> SQL: check ProjectMember / Project
  -> ingest queue
     -> SQL: insert ingest_jobs(status=pending)
     -> SQL: update ingest_jobs(status=processing, progress)
  -> auto_ingest()
     -> parse_document()
     -> check_cache()
     -> load_checkpoint()
     -> read purpose/index/overview/schema
     -> truncate source text
     -> LLM analysis
     -> save checkpoint(step=1)
     -> SQL: update ingest_jobs(step=1)
     -> LLM generation
     -> save checkpoint(step=2)
     -> SQL: update ingest_jobs(step=2)
     -> parse FILE blocks
     -> write_file_blocks()
        -> optional LLM merge for existing pages
     -> SQL: update ingest_jobs(step=3)
     -> embed_pages()
        -> LanceDB: delete old page chunks + add new vectors
     -> update_cache()
     -> remove_checkpoint()
  -> SQL: update ingest_jobs(status=done, files_written, completed_at)

24. 如果你要继续优化 ingest，应该先看哪里

  1. app/ingest/pipeline.py：决定主流程、截断方式、checkpoint 和上下文装配
  2. app/ingest/prompts.py：决定 analysis / generation / merge 的行为边界
  3. app/ingest/writer.py：决定已有页面怎么合并、哪些页面特殊处理
  4. app/documents/parser.py：决定原始文档被解析成什么文本
  5. app/ingest/parser.py：决定 LLM 输出格式容错和路径安全

25. 简短总结

  • 它不是 Agent
  • 它是固定的两阶段 LLM 编译流水线
  • 原始文档文本会直接进入 prompt，但最多只取前 100000 字符
  • 输出必须是严格的 FILE block
  • 已有页面会在写盘阶段再次做 LLM merge
  • SQL 业务库只保存项目、权限和 job 状态，不保存 wiki 正文
  • LanceDB 只在 embedding 阶段被写入，用于后续向量检索
  • 队列支持同项目串行、失败重试、服务重启恢复

如果后续要把 ingest 改成 Agent 或 toolcall 架构，这份文档描述的就是当前基线行为。
